Vestigant Spotlight v0.9.20 change / validation note

Purpose
- Follow-up to V0_9_18 reused-cache iOS run, which built successfully but still hit the 5 GiB DB guardrail at ~150,000 parsed CoreSpotlight records.
- V0_9_18 showed the remaining bloat was not only raw_key_values/date rows. The active case DB was already ~2.1 GiB immediately after importing the reused iOS FFS inventory and app database inventory, before native CoreSpotlight parsing substantially expanded the DB.

Observed V0_9_18 failure class
- Build succeeded as Vestigant Spotlight v0.9.18.
- Reuse-cache run reached native_parse_start and then failed_sqlite_size_guardrail.
- At guardrail hit: parsed_items=150000, raw_key_values=1405771, raw_date_candidates=463449, DB=5479522304 bytes, WAL=0.
- The run later logged failed_exception: sqlite exec failed: cannot commit - no transaction is active. Cause: the guardrail exception was caught as a per-item/per-store parse failure, so parsing continued with transaction state already changed.

V0_9_20 fixes
1. Normal iOS Spotlight-first mode no longer materializes full FFS inventory rows into the active case DB by default.
   - Cached FFS inventory is referenced/counted and remains in the reuse-cache source, but millions of file rows are not inserted into the new active case DB.
   - New support flag: --materialize-ios-ffs-inventory / --full-ios-ffs-inventory-db.

2. Normal iOS Spotlight-first mode no longer parses and persists broad app database record rows by default.
   - App database inventory is still retained, but broad parsed app records are opt-in.
   - New support flag: --materialize-ios-app-db-records / --full-ios-app-db-records.
   - Convenience support flag: --materialize-ios-support-db enables both full FFS inventory and full app DB parsed-record materialization.

3. Default iOS raw_key_values persistence is now reference-only.
   - Broad displayName/title/name/date/ranking/metadata property rows are decoded for derived review where possible but not stored one-row-per-property in raw_key_values.
   - Default raw_key_values stores only strong forensic-reference values, such as URL/path/container/account/contact/message identifiers, with a maximum of two rows per record.
   - Full raw native key/value persistence remains opt-in with --diagnostic-full-native-db.

4. Default iOS raw_date_candidates persistence is reduced.
   - raw_records.last_updated_utc remains the compact per-record index/update timestamp.
   - Last_Updated is no longer duplicated into raw_date_candidates for every iOS record in normal mode.
   - Normal mode persists at most one clearly high-value activity/content date per record; full date expansion remains diagnostics-only.

5. DB guardrail is now fatal instead of being swallowed as item/store parse failure.
   - Guardrail exceptions are rethrown out of metadata item and store catches.
   - This avoids the V0_9_18 secondary error: cannot commit - no transaction is active.

Expected V0_9_20 result
- Reused-cache iOS run should keep DB/WAL materially lower than V0_9_18 because it avoids the initial full FFS/app DB materialization and sharply reduces raw_key_values/raw_date_candidates.
- If growth still exceeds guardrail, the run should fail cleanly at failed_sqlite_size_guardrail without the misleading follow-on COMMIT error.

Validation performed here
- Linux CMake configured successfully. The modified native parser and case database sources compiled; the environment then timed out while compiling the very large app_runner.cpp translation unit. The Windows-only GUI source was not compiled in this Linux environment, so full MSVC validation remains required.
- Full MSVC validation still needs to be run on the Windows build host.

V0_9_20 additional GUI/context fix:
- iOS Missing From FFS Candidates now includes spotlight_text_context_sample and spotlight_text_context_status where compact same-record text was recovered from the Spotlight index.
- The parser writes at most one synthetic __spotlight_investigator_text_context value for records that already had a persisted file/path/URL/account reference, preserving investigative context without restoring broad raw property materialization.
- Referenced-path extraction explicitly excludes the synthetic text-context row so context text cannot create false path candidates.
