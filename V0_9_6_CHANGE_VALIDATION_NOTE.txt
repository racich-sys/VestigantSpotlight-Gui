Vestigant Spotlight v0.9.6 Change and Validation Note

Baseline
- Started from user-uploaded v0.9.5 source/build/runtime results.
- v0.9.5 build log showed a successful MSVC build and binary version Vestigant Spotlight v0.9.5.
- v0.9.5 reused the v0.9.4 large iOS FFS cache successfully, but runtime status showed a long gap between open_sqlite and ios_ffs_inventory_import while importing 1,592,440 FFS inventory rows and 11,248 app database inventory rows from cached CSV.

Problem addressed
- v0.9.5 avoided relisting/reextracting the 257 GiB ZIP, but it still re-imported large inventory CSV files through an in-memory readCsv path.
- This preserved correctness but still caused long runtimes and high memory pressure during reuse-cache testing.

Changes in v0.9.6
1. Added a cache-SQLite fast path for --reuse-ios-cache.
   - When the prior cache case contains VestigantSpotlight.case.sqlite or spotlight_case.db, v0.9.6 ATTACHes that database and copies ios_ffs_file_inventory and ios_app_database_inventory rows into the new case with the new source_id.
   - This avoids reparsing the multi-hundred-MB cached CSVs during reuse tests.
   - New run status messages include ios_ffs_inventory_cache_db_import_start and ios_ffs_inventory_cache_db_import_complete.

2. Added streaming CSV fallback.
   - If the prior cache database is missing, locked, incompatible, or missing required tables, v0.9.6 falls back to streaming CSV import.
   - The fallback no longer loads all CSV rows into memory before insert.
   - New progress messages include ios_ffs_inventory_csv_stream_import_start, ios_ffs_inventory_csv_stream_progress, and ios_app_database_inventory_csv_stream_progress.

3. Preserved forensic caution.
   - The source cache manifest and reuse log still identify the reused cache case and original source ZIP path.
   - Cache reuse remains a working-copy acceleration. The prior cache folder must remain available because extracted app database paths can point back to the prior case staging folder.
