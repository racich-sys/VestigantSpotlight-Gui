Vestigant Spotlight V0_9_4 change/validation note

Baseline: V0_9_3 uploaded source package and build log.

Primary fix:
- Large iOS FFS ZIP staging no longer builds a complete in-memory ArrayList of every 7-Zip entry before writing ios_ffs_file_inventory.csv.
- The generated iOS focused ZIP extraction script now streams 7z l -slt output into the FFS/app database inventory CSVs and flushes progress every 10,000 entries.
- Added logs/ios_zip_stage_heartbeat.log and logs/ios_zip_inventory_progress.tsv so a 250 GB ZIP does not appear silent/hung during stage_zip_source.
- Targeted app database extraction is started before the full FFS inventory stream, and duplicate extraction is skipped.
- Focused CoreSpotlight extraction now writes heartbeat lines before and after the 7z stage.

Expected operational effect:
- During very large ZIP intake, investigators should see timestamps advance in ios_zip_stage_heartbeat.log, ios_zip_inventory_progress.tsv, ios_ffs_file_inventory.csv, or ios_app_database_inventory.csv.
- If the process stalls, the last heartbeat now identifies whether it was targeted app database extraction, focused CoreSpotlight extraction, or streaming FFS inventory.

Validation performed in this environment:
- Static patch applied from V0_9_3.
- Linux CMake/core build and self-test should be run after packaging; Windows/MSVC and GUI runtime validation must be run on the user workstation.

Recommended user validation:
1. Build with build_windows_msvc.bat.
2. Run VestigantSpotlightCli.exe --version and confirm v0.9.4.
3. Run VestigantSpotlightTests.exe.
4. Re-run the 250 GB WhatsApp-containing iOS FFS.
5. Monitor Q:\SpotlightCase\<case>\logs\ios_zip_stage_heartbeat.log and ios_zip_inventory_progress.tsv while stage_zip_source is active.
